# opencode-autopolit

Фреймворк принудительного рабочего процесса - Исследование→Спецификация→Верификация→Разработка(TDD)→Проверка→Архивирование

[English](README.md) | [简体中文](README.zh-cn.md) | [日本語](README.ja.md) | [한국어](README.ko.md) | **Русский**

## Обзор

Интегрированная среда разработки на основе oh-my-openagent + OpenSpec + Superpowers. Принудительное выполнение спецификационно-ориентированного и тесто-ориентированного процесса разработки.

## Основной рабочий процесс

```
EXPLORE → SPEC → DESIGN VERIFICATION → DEV → VERIFY → ARCHIVE
   ↓       ↓           ↓              ↓       ↓         ↓
  CRG   Proposal    Проверка       TDD    Checklists   Merge
       Specs/Design              RED→GR→RF  User Approval
       (BLOCKS DEV)
```

## Сравнительный анализ фреймворков

### Проблемы существующих решений

| Фреймворк | Ключевое преимущество | Ключевой недостаток |
|-----------|----------------------|---------------------|
| **oh-my-openagent** | Мощная система плагинов, hooks, оркестрацияworkflow | Не хватает спецификационно-ориентированногоworkflow, легко "писать как думается" |
| **OpenSpec** | Система документации (proposal→specs→design→tasks) | Только инструмент документации, нет исполнения, можно пропустить |
| **Superpowers** | Библиотека навыков (TDD, debugging, code review) | Требует ручного вызова, Agent часто игнорирует или забывает |
| **Code-Review-Graph** | Анализ структуры кода, зависимостей, радиуса влияния | "После уроков", не может предотвратить ошибки проектирования |
| **WeChat-ACP** | Уведомления WeChat, синхронизация прогресса | Только уведомления, нет интеграцииworkflow |

### Ключевые противоречия

1. **Спецификация vs Произвол**: OpenSpec предоставляет спецификацию, но Agent может пропустить и сразу писать код
2. **Проектирование vs Реализация**: При написании кода предполагается существование модулей, при реализации выясняется что их нет
3. **Инструменты vs Поведение**: Множество хороших инструментов, но Agent не обязательно их использует

### Решение: Принудительный рабочий процесс

```
┌─────────────────────────────────────────────────────────────────┐
│                     opencode-autopolit                          │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────┐   ┌─────────┐   ┌──────────────┐   ┌───────────┐  │
│  │  CRG    │ + │OpenSpec │ + │DESIGN        │ + │Superpowers│  │
│  │Анализ   │   │Документ │   │VERIFICATION  │   │Выполнение │  │
│  └────┬────┘   └────┬────┘   └──────┬───────┘   └─────┬─────┘  │
│       │            │              │                  │         │
│       ▼            ▼              ▼                  ▼         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │           Оркестратор принудительного workflow          │   │
│  │  EXPLORE → SPEC → DESIGN VERIFY → DEV → VERIFY → ARCHIVE │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### Почему мы так делаем

| Проблема | Решение | Результат |
|----------|---------|-----------|
| Предполагается существование модуля, но его нет | DESIGN VERIFICATION проверяет все предположения | Ошибки обнаружены до компиляции |
| Пропуск TDD и直接ное написание кода | tdd-phase принуждает RED→GREEN→REFACTOR | Повышение покрытия тестами |
| Спецификации игнорируются | Интеграция в workflow, каждый этап обязателен | 100% реализация спецификаций |
| Инструменты работают отдельно | Единый триггер (keyword-detector) | Координация инструментов |
| Ретроспективный код Review | CRG анализирует заранее на этапе EXPLORE | Профилактика лучше лечения |

### Ключевая инновация: DESIGN VERIFICATION

Это решение проблемы "AI галлюцинаций в коде":

```
Предположения при написании кода:
✗ import from 'shared/types'       → Реальный путь: '@shared/types'
✗ use createHook()                 → Реальный API: createHooks()
✗ class UserService exists         → Реально: UserAPI

Что делает DESIGN VERIFICATION:
1. Парсит все import-выражения
2. Проверяет существование путей модулей
3. Верифицирует соответствие имен API
4. Проверяет существование типовых определений
5. Генерирует отчет, блокирует DEV до проверки всех предположений
```

## Использование

### Автоматический триггер

При вводе ключевых слов workflow запускается автоматически:

| Ключевое слово | Триггер |
|----------------|---------|
| `implement xxx` | ✅ |
| `add xxx` | ✅ |
| `create xxx` | ✅ |
| `fix xxx` | ✅ |
| `build xxx` | ✅ |
| `tdd` | ✅ |
| `test-first` | ✅ |
| `refactor` | ✅ |

### Ручной триггер

Также можно использовать slash command (если настроен соответствующий skill):

```
/autopolit implement user login
```

## Этапы рабочего процесса

### 1. EXPLORE - Этап исследования
- Анализ кодовой базы с помощью CRG (Code-Review-Graph)
- Генерация списка предположений (assumption list)

### 2. SPEC - Этап спецификации
- Генерация proposal → specs → design → tasks
- Гранулярность документов 2-5 минут

### 3. DESIGN VERIFICATION - Этап верификации (новое)
- Проверка всех предположений: пути модулей, типы, API
- **Блокировка DEV до подтверждения всех предположений**

### 4. DEV - Этап разработки
- Принудительный TDD: RED (тест) → GREEN (реализация) → REFACTOR
- Запрещено писать код без тестов

### 5. VERIFY - Этап проверки
- Чек-лист: нет TODO, нет заглушек, нет AI-slop
- Подтверждение пользователя

### 6. ARCHIVE - Этап архивирования
- Слияние delta specs в основную ветку

## Результат триггера

После ввода `implement user login`, Agent получит:

```
🤖 AUTOPOLIT WORKFLOW ACTIVATED

🔄 WORKFLOW SEQUENCE (MANDATORY):
1. EXPLORE → 2. SPEC → 3. DESIGN VERIFICATION → 4. DEV → 5. VERIFY → 6. ARCHIVE
```
